## L1s vs L2s

The terms L1 and L2 describe **where execution and settlement happen** in a blockchain stack.

An **L1 (Layer 1)** is a base blockchain that:
- Maintains its own consensus mechanism.
- Has its own validator set that independently agrees on the canonical chain.
- Settles transactions directly on its own ledger — there is no parent chain it "reports back to" for finality.
- Examples: Avalanche, Ethereum, Bitcoin, Solana.

An **L2 (Layer 2)** is a system that:
- Processes transactions off the main chain to increase throughput or reduce cost.
- Relies on an L1 for final settlement and security. The L2 periodically posts proofs or compressed transaction data back to the L1.
- Examples: Optimism, Arbitrum, zkSync (all posting to Ethereum).

Avalanche's custom chains are **true L1s** in this sense: each one has its own validator set running independent consensus, not a rollup posting proofs to another chain. They benefit from Avalanche's consensus protocol but are not dependent on the C-Chain for finality. This gives custom L1s much higher throughput limits and lower latency than L2 rollup approaches.

The distinction matters when advising builders: if they want full sovereignty, custom gas tokens, and maximum performance, an Avalanche L1 is the right architecture. If they want maximum liquidity and composability with Ethereum's existing ecosystem and are willing to pay rollup overhead, an Ethereum L2 may be better.

## What is a Node?

A **node** is a computer running the Avalanche client software (AvalancheGo) that participates in the network. Nodes perform two related roles:

- **Validator**: a node that has staked AVAX (on the primary network) and is responsible for participating in consensus. Validators vote on the validity of transactions and the canonical ordering of blocks.
- **Non-validating (RPC) node**: a node that tracks the chain state and answers API queries but does not participate in consensus voting. These are used for read-heavy applications.

For an Avalanche custom L1, the validators are the specific nodes that have been registered as validators for that particular blockchain. They run their own consensus rounds independently of the rest of the Avalanche network.

### Minimum Nodes to Run a Blockchain

For production mainnet deployments, Avalanche guidance is to run at least **5 validators** for security and liveness. A chain can run with fewer in development/testing environments, but with too few validators the network becomes fragile: if one node goes offline, the chain can stall.

In practice, production deployments use significantly more validators to ensure decentralization and fault tolerance. Many live L1s run with 10–20 validators, and larger networks run with many more.

### Minimum Uptime for Node Rewards

For **Primary Network** validators, published mainnet parameters currently document an **uptime requirement of 80%** over the staking period to remain eligible for staking rewards (see the [rewards formula](https://build.avax.network/docs/primary-network/validate/rewards-formula) docs). **[ACP-267](https://docs.avax.network/docs/acps/267-uptime-requirement-increase)** proposes raising that threshold to **90%**; requirements can change with network upgrades, so always confirm the live value in current Avalanche documentation before advising operators.

Validators that fall below the applicable uptime threshold receive no rewards for that period. They are not slashed (their stake is not taken), but they forfeit income.

## Roles: Admin, Manager, and Enabled

Avalanche precompiles and access-control contracts use a three-tier permission model to separate different levels of authority:

**Admin**: the highest privilege level. An admin can add or remove other admins, add or remove managers, and add or remove enabled addresses. Admins can also directly perform any action available to lower tiers. The admin set should be tightly controlled — typically a multisig or governance contract.

**Manager**: a mid-tier role introduced in more recent versions. A manager can add or remove enabled addresses but cannot change the admin or manager set. This allows operational teams to manage day-to-day access without having root control over the system.

**Enabled**: addresses that are permitted to perform the specific action controlled by the precompile (e.g., deploy contracts, mint native tokens, submit transactions). They cannot modify any access lists.

This role hierarchy follows the principle of least privilege: you grant the minimum permissions needed for each use case, limiting the blast radius of a compromised key.

## What is AWM?

**Avalanche Warp Messaging (AWM)** is a native, trust-minimized protocol for passing arbitrary messages between Avalanche blockchains. It works by having the sending chain's validators co-sign a message using BLS cryptography, producing an aggregated signature. The receiving chain's validators verify that signature against the registered validator set without relying on any bridge contract or external oracle.

This is fundamentally different from "wrapped bridge" approaches where a centralized entity or a multisig controls the funds crossing the bridge. With AWM, the security is grounded in the consensus layer itself — the same validators that produce blocks also vouch for cross-chain messages.

AWM is the foundation for **Teleporter**, which provides a higher-level API for cross-chain messaging. **ICTT** (Interchain Token Transfer) is then built on top of that messaging stack for token bridging flows. A technical Team1 member should understand AWM as the primitive, Teleporter as the messaging layer abstraction, and ICTT as a token-transfer application layer.

## Precompiles

A **precompile** is a smart contract with a well-known address whose logic is implemented directly in the Avalanche node software (Go code), not in Solidity bytecode. From the Solidity developer's perspective, precompiles look like normal contracts — you call them at a fixed address — but they execute natively, without EVM interpretation, making them far more gas-efficient and capable of operations the EVM cannot otherwise perform (like accessing validator sets or aggregating BLS signatures).

Avalanche's custom L1s are configured through a set of precompiles enabled in the genesis file. Here is what each one does:

### `deployerAllowListConfig`

Controls **who can deploy smart contracts** on the chain. When enabled, only addresses in the "enabled" (or higher) list can submit deployment transactions (transactions with a blank `to` field). All other addresses receive a revert if they attempt to deploy.

This is essential for permissioned environments: a gaming company running its own L1 might want only its own backend to deploy contracts, preventing third parties from deploying arbitrary logic on the chain.

```json
"deployerAllowListConfig": {
  "blockTimestamp": 0,
  "adminAddresses": ["0xAdminAddress"],
  "enabledAddresses": ["0xDeployerAddress"]
}
```

### `nativeMinterConfig`

Controls **who can mint the chain's native token**. When enabled, only authorized addresses can call the precompile's `mintNativeCoin(address, amount)` function to create new native currency out of thin air.

This powers app-specific tokenomics: a game chain might mint in-game currency as rewards for players, or a DeFi chain might have an inflation mechanism managed by a DAO contract.

```json
"nativeMinterConfig": {
  "blockTimestamp": 0,
  "adminAddresses": ["0xAdminAddress"],
  "enabledAddresses": ["0xMinterContract"]
}
```

### `txAllowListConfig`

Controls **who can submit any transaction** on the chain. Even more restrictive than `deployerAllowListConfig`, this limits transaction submission to a whitelist of addresses. Only enabled addresses can send any transaction at all — regular transfers, contract calls, and deployments.

Use cases: enterprise chains where only known participants may interact, KYC-gated chains where users must be verified before transacting, and testing environments.

```json
"txAllowListConfig": {
  "blockTimestamp": 0,
  "adminAddresses": ["0xAdminAddress"],
  "enabledAddresses": ["0xUser1", "0xUser2"]
}
```

### `feeManagerConfig`

Controls **dynamic fee configuration** at runtime. Once enabled, authorized addresses can adjust fee parameters (base fee, block gas limit, target block rate) without redeploying the chain. This allows fee management to respond to changing network conditions or business requirements.

Without `feeManagerConfig`, fee parameters are fixed at genesis and can only change through a hard fork. With it, a governance contract can vote to adjust fees and execute the change in a single transaction.

```json
"feeManagerConfig": {
  "blockTimestamp": 0,
  "adminAddresses": ["0xFeeGovernanceContract"]
}
```

### `rewardManagerConfig`

Controls **where block rewards go** and whether fee burning is enabled. Options include:
- Burning all fees (deflationary; no validator rewards from fees).
- Sending fees to validators (validator rewards from fees).
- Sending fees to a fixed address (e.g., a treasury or redistribution contract).

This precompile lets the chain's governance decide in real time how value flows from transaction fees — a key lever for custom tokenomics.

```json
"rewardManagerConfig": {
  "blockTimestamp": 0,
  "adminAddresses": ["0xRewardAdmin"],
  "initialRewardConfig": {
    "allowFeeRecipients": true
  }
}
```

### `warpConfig`

Enables **Avalanche Warp Messaging** on the chain. This is the gateway precompile for cross-chain communication. It exposes:
- `sendWarpMessage(bytes calldata payload)`: emits a message that validators sign.
- `getVerifiedWarpMessage(uint32 index)`: reads a verified warp message from the block (after validators have verified it off-chain and embedded the result in the block header).

Enabling `warpConfig` is a prerequisite for using Teleporter (ICTT) or any other cross-chain protocol on your L1.

```json
"warpConfig": {
  "blockTimestamp": 0,
  "quorumNumerator": 67
}
```

The `quorumNumerator` defines the percentage of stake that must co-sign a warp message for it to be considered valid (default 67 = two-thirds).

## Examples of Custom L1s

The Avalanche ecosystem includes many live custom L1s. The snapshots below are deliberately detailed: they show how sovereignty shows up in practice. Avalanche is not just “a chain for one app.”

### Beam (gaming, and the move to sovereign L1 validation)

[Beam](https://docs.onbeam.com/chain/builder/network-overview) is a gaming-focused network on Avalanche (EVM-compatible) used by studios and players for on-chain game economies, marketplaces, and adjacent tooling. For builders tracing IDs in explorers and RPC configs, Beam still publishes **Subnet ID**, **VM ID**, and **Blockchain ID** for mainnet and testnet—the same logical chain can be discussed as an Avalanche L1 under the post-Etna vocabulary.

What might matter for Team1 conversations is **why Beam talks about a migration**: Avalanche’s **Etna** upgrade and related changes (including **[ACP-77](https://docs.avax.network/docs/acps/77-reinventing-subnets)**) rework how dedicated networks run: L1s can have their **own validator set and fee model** without tying every validator to the same primary-network staking rules that applied to classic subnets. That shift is exactly the kind of “sovereignty upgrade” chains want when they need predictable costs, custom parameters, and a path toward **broader participation in securing the network**.

Beam’s public roadmap messaging adds another layer: the **Horizon** upgrade is described as introducing a **permissionless proof-of-stake** path where community members can run or delegate to **Beam Nodes**, with incentives aligned to the **BEAM** token and activity across Beam-native products (for example DEX, bridge, and marketplace surfaces in their ecosystem). Whether you are advising a game studio or operating nodes, the lesson is the same: **custom L1s are not frozen at launch**, they evolve with Avalanche protocol upgrades and the project’s own governance.

### Gunzilla, GUNZ, and *Off The Grid*

[GUNZ](https://gunbygunz.com/) is Gunzilla Games’ **gaming economy chain** on Avalanche, announced as a customized subnet designed so gameplay can rely on **low-latency execution** and a **gas token aligned with the game economy** (rather than a shared public mempool priced for unrelated DeFi traffic). Gunzilla’s own launch materials describe GUNZ as built to feel natural to both Web3 and traditional players: **optional** on-chain ownership (for example **Play-to-Own** and NFT-based items) without forcing every player through wallet friction on day one.

The flagship integration is ***Off The Grid*** (*OTG*), a AAA-style battle royale where narrative and progression intersect with digital ownership—Gunzilla positions GUNZ as the settlement layer for that economy first, with **SDK-style building blocks** (companion app, marketplace, minting, and related infrastructure) so other studios can ship on the same stack. For Avalanche operators, GUNZ is a textbook example of **app-specific fee assets**, **permissioning tuned for publishing** (studios care deeply about who can deploy what), and **performance requirements** that are hard to meet on a general-purpose shared chain.

### FIFA Blockchain and the FIFA Collect migration

In 2025 FIFA announced the **FIFA Blockchain**, a **custom Avalanche L1** aimed at global-scale digital products—starting with **[FIFA Collect](https://collect.fifa.com/)**, the federation’s official collectibles platform. Avalanche’s announcement emphasizes why an institution chooses an L1: **sub-second finality**, **low fees**, and **room to grow** without a single shared chain becoming the bottleneck during high-traffic moments (think tournament drops and major campaigns). The chain is **EVM-compatible**, which matters for **wallet support** (for example MetaMask and WalletConnect flows) and for reusing standard tooling.

The **migration** itself is instructive for anyone advising enterprises. FIFA Collect’s [migration FAQ](https://collect.fifa.com/pages/migration-faq) spells out user-visible constraints that always surface in real deployments: a planned **maintenance window** (on the order of **~12 hours**), **no fee to move existing collectibles**, a shift from **non-EVM wallets** (for example prior Algorand-based wallets) to **EVM wallets** after cutover, and policy details around **balances** and **listings** that must be handled deliberately. **[Modex](https://modex.tech/)** is named as the long-running FIFA Collect partner leading marketplace and infrastructure work—another reminder that **sovereign L1s still ship with operator partners** responsible for product, compliance, and continuity.

### Dexalot (central limit order book on a dedicated chain)

[Dexalot](https://avax.network/about/blog/dexalot-launches-first-central-limit-order-book-subnet-on-avalanche-with-1m-in-incentives) is a decentralized exchange that runs a **central limit order book (CLOB)** on its **own Avalanche deployment** rather than competing for block space with unrelated contracts on the C-Chain. The architecture is deliberately **split**: users typically **deposit and withdraw** assets on the **C-Chain** (where liquidity and bridge endpoints already exist), while **matching, cancels, and trading loops** execute on Dexalot’s dedicated chain with its own **native gas token (ALOT)** and chain parameters tuned for **high-frequency order flow**.

Why this matters technically: CLOB matching wants **deterministic ordering**, **tight latency**, and **predictable cost per cancel/replace**—workloads that degrade quickly when a single chain is also processing NFT mints, long-tail DeFi, and arbitrary contracts. Dexalot’s write-ups highlight **price–time priority** and **low per-action fees** as first-class goals—exactly the sort of outcome custom L1s exist to provide.

### SK Planet and UPTN (enterprise loyalty at national scale)

[SK Planet](https://www.skplanet.com/eng/business/blockchain), a major South Korean digital commerce and platform company, worked with **[AvaCloud](https://www.avacloud.io/)** to launch **UPTN**, a **dedicated Avalanche L1** backing Web3 features for SK’s consumer products. Avalanche’s [case study](https://avax.network/case-studies/how-sk-planet-revolutionized-customer-loyalty-on-chain) on the program is unusually concrete: SK started with **OK Cashbag**, a loyalty program with **tens of millions** of users and **tens of thousands** of retail partners, and reported **over 100,000** new sign-ups in the first **48 hours** after Web3-enabled rewards went live—evidence that **infrastructure must survive flash crowds**.

Two enterprise patterns show up clearly. First, a **Web2.5** rollout: upgrade an existing product so most users keep their familiar flows while new benefits (for example **NFT-based rewards** and stronger ownership semantics) appear gradually. Second, **operational UX**: AvaCloud’s write-up calls out a **gasless relayer**-style capability so end users are not stuck paying gas in a loyalty context—something platform teams almost always ask for. SK’s roadmap also ties the same L1 directionally to **NFT ticketing** (including work with **Dreamus**) where on-chain authenticity and secondary-market rules matter as much as raw TPS.


Each of these networks benefits from the same underlying idea: **full control over gas token, fee policy, permissioning, and block production**, while remaining able to interoperate with the wider Avalanche ecosystem through **AWM**, **Teleporter**, and **ICTT** where builders enable them.

## Key Takeaways

- Avalanche L1s are sovereign blockchains with their own validator sets and consensus. They differ from L2s, which depend on a parent chain for security and finality.
- A node is a computer running AvalancheGo; a Primary Network validator must satisfy the network’s published uptime requirement (see current docs; **80%** in listed mainnet parameters, with **ACP-267** proposing **90%**) to earn rewards.
- For production deployments, at least 5 validators are strongly recommended to run a resilient custom L1.
- The admin/manager/enabled role hierarchy separates governance authority from operational authority, following the principle of least privilege.
- AWM is the native cross-chain messaging primitive; it uses BLS aggregate signatures from validators to verify messages without a centralized bridge.
- Precompiles are native contracts (implemented in Go) enabled at genesis that extend the EVM with capabilities like contract deployment control, native token minting, transaction allow-lists, dynamic fees, configurable rewards, and cross-chain messaging.
- Live examples—Beam’s L1 validation roadmap, Gunzilla’s GUNZ for *Off The Grid*, FIFA’s dedicated chain and Collect migration, Dexalot’s CLOB L1, and SK Planet’s UPTN loyalty deployment—show why app-specific sovereignty matters: dedicated block space, migrations you can plan for, custom tokenomics, and permissioning that matches the product.
